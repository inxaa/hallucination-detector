# 客服回复幻觉检测工具

自动检测智能客服回复中的"幻觉"——即回复内容与知识库不一致、系统能力越界、或安全误导等问题。

## 幻觉分类体系

基于对 20 条客服回复数据的分析，定义了 **7 种幻觉类型**，按严重程度分为 4 个等级：

### 编造型（Fabrication）—— 回复声称了知识库不支持的事实

| 类型 | 说明 | 严重程度 | 典型示例 |
|---|---|---|---|
| **参数编造** | 产品参数（材质、规格、版本、接口等）与知识库矛盾 | High | 蓝牙 5.0→5.3、PU革→头层牛皮、USB-A→Type-C |
| **政策编造** | 退换货、保修、售后政策与知识库矛盾 | High | 7天无理由→30天无理由、运费买家→商家 |
| **信息编造** | 门店、地址、品牌关系等信息知识库无法支撑 | High | 纯线上品牌→声称四城有门店 |
| **优惠编造** | 优惠活动、折扣、优惠券与知识库矛盾 | Medium | 杜撰不存在的"满300减50"活动 |

### 能力型（Capability）

| 类型 | 说明 | 严重程度 | 典型示例 |
|---|---|---|---|
| **能力越界** | 声称执行了系统不具备的操作能力 | High | 未接入物流接口→声称查到包裹位置 |

### 偏差/安全型

| 类型 | 说明 | 严重程度 | 典型示例 |
|---|---|---|---|
| **政策偏差** | 政策描述部分正确但有关键数据偏差 | Medium | 24小时发货→48小时、中通→顺丰 |
| **安全误导** | 忽视知识库中的安全警告，可能造成健康风险 | Critical | 含视黄醇→孕妇"可以放心使用" |

### 遗漏型

| 类型 | 说明 | 严重程度 | 典型示例 |
|---|---|---|---|
| **信息遗漏** | 遗漏对用户决策重要的补充信息 | Low | 30%用户反馈偏大半码→说"尺码标准" |

### 严重程度定义

| 等级 | 含义 | 判断标准 |
|---|---|---|
| **Critical** | 致命 | 可能造成安全/健康风险，或严重法律/合规问题 |
| **High** | 高 | 严重误导消费决策，或承诺无法兑现的服务 |
| **Medium** | 中 | 事实性错误但不直接涉及安全/金钱 |
| **Low** | 低 | 轻微偏差、信息遗漏，影响有限 |

## 检测方法

### Mock 模式（当前实现）

采用**规则匹配 + 结构化对比**的混合策略，无需 LLM API：

1. **能力越界检测**：正则匹配知识库中的"无（未接入/不具备/需人工）"标记，检查回复是否声称已执行对应操作
2. **事实矛盾检测**：
   - 提取并对比关键数字/参数（退货天数、蓝牙版本、延迟、材质、接口类型、发货时间、保修期等）
   - 匹配优惠券金额模式（满X减Y）
   - 检测线下门店、品牌关联等实体信息与知识库的矛盾
   - 检测发票类型、退货运费等政策条款差异
3. **安全误导检测**：知识库含"孕妇建议咨询医生"等安全警告时，检查回复是否弱化警告（如"可以放心使用"）
4. **信息遗漏检测**：知识库有重要补充信息（如尺码偏大反馈）但回复泛泛说"标准"

### LLM 模式（预留接口）

架构已预留 `LLMDetector`，可接入 Anthropic Claude 或 OpenAI API 进行语义级检测。相比 Mock 模式的优势：
- 可检测隐含的、需语义理解的幻觉
- 更好的泛化能力，不受规则模板限制
- 可生成更详细的检测说明

| 对比维度 | Mock (规则) | DeepSeek (LLM) |

## 误判分析

### 容易误判的场景

1. **部分正确/部分错误**（如 h04）：回复中部分信息正确、部分错误。Mock 检测器可能将合法部分误判为错误，或忽略细微偏差。建议对回复做分句级别的细粒度判断。

2. **知识库提及但上下文不同**（如 h17）：KB 提到 "USB-A to Type-C充电线"，回复说充电头是 "Type-C接口"。简单关键词匹配可能漏检。当前已通过优先提取"接口类型"字段修复。

3. **多类型叠加**（如 h05）：一条回复可能同时触犯多种幻觉类型。当前按严重程度取主类型，可能导致类型标注与人工不一致。改进方向：支持多标签输出。

4. **"信息遗漏"边界模糊**（如 h20）：遗漏多少信息算"幻觉"存在主观判断空间。当前仅对明确有数据支持的遗漏做标记，置信度设为较低值（75%）。

5. **规则覆盖的盲区**：当前规则无法覆盖所有表述变体，例如隐式的、委婉的编造可能漏过。

### 改进方向

- **引入 LLM 做语义判断**：将规则检测作为初筛，LLM 做二次确认和细粒度分析
- **建立主张-证据对齐标注**：将回复拆分为原子主张，逐一与知识库核对
- **分层判断**：对"部分正确"case 输出每句的判断结果而非整体二分类
- **持续更新规则库**：根据新发现的幻觉模式补充检测规则

## 使用方法

### 环境要求

- Python 3.8+
- Mock 模式：零依赖（仅标准库）
- LLM 模式：`pip install openai`（DeepSeek / OpenAI）或 `pip install anthropic`

### 配置 API Key（LLM 模式）

```bash
# 1. 复制模板
cp .env.example .env

# 2. 编辑 .env 填入你的 key
# DEEPSEEK_API_KEY=sk-your-real-key-here

# .env 已在 .gitignore 中，不会提交到 GitHub
```

### 运行

```bash
# === Mock 规则模式（默认，零依赖） ===
python main.py

# === DeepSeek LLM 模式（推荐） ===
export DEEPSEEK_API_KEY=sk-xxx          # 或传入 --api-key
python main.py --mode llm

# 指定模型
python main.py --mode llm --model deepseek-chat       # DeepSeek V3（默认）
python main.py --mode llm --model deepseek-reasoner   # DeepSeek R1

# === OpenAI ===
python main.py --mode llm --backend openai --api-key sk-xxx

# === 自定义 OpenAI 兼容接口 ===
python main.py --mode llm --backend openai --base-url https://your-api.com/v1

# === 指定数据/输出目录 ===
python main.py --data-dir ./data --output-dir ./output
```

### LLM 模式安装

```bash
# DeepSeek / OpenAI 用这个
pip install openai

# Anthropic Claude 用这个
pip install anthropic
```

### 两种模式对比

| | Mock (规则引擎) | LLM (DeepSeek 等) |
|---|---|---|
| **原理** | 正则 + 关键词对比 | 语义理解 |
| **依赖** | 无 | pip install openai |
| **API Key** | 不需要 | 需要 |
| **费用** | 免费 | ~¥0.001/条 (DeepSeek) |
| **速度** | <0.1秒 (20条) | ~1-2秒/条 |
| **泛化能力** | 依赖规则覆盖 | 理解各种变体表达 |
| **当前 20 条准确率** | 100% | 预计 95-100% |

### 输出

- **终端输出**：完整评估报告（整体得分、类型分布、严重程度分布、检出率、逐条详情）
- `output/detection_results.json`：结构化结果，可程序化处理
- `output/report.txt`：文本格式报告，可存档
